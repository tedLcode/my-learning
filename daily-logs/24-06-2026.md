## 📚 What I Studied Today
- Topic: Service Accounts in Enterprise BI Infrastructure
  - What I understood: Service accounts are non-personal accounts used by automated systems (scheduled refresh jobs, deployment pipelines) rather than individual users — they need to exist before any automation can be configured on top of the Power BI workspace
  - What I understood: Creating separate service accounts for UAT and PROD is essential — using the same account across environments would mean a PROD credential is exposed in the UAT environment, which is a security and governance concern

- Topic: Git Integration for Power BI
  - What I understood: Power BI's Git integration allows report definitions and dataset schemas to be version-controlled, enabling change tracking and CI/CD-style promotion from UAT to PROD
  - What I understood: Before Git integration can be configured, either a new repository must be created in the correct org, or access to an existing one must be granted — this step requires coordination with the admin/platform team, hence the current blocker

- Topic: Dependency Mapping in Infrastructure Work
  - What I understood: Infrastructure tasks rarely have a single critical path — service accounts and Git setup can happen in parallel, but Git access is blocked externally while service accounts are an internal action; recognizing this distinction helps prioritize what to act on vs. what to track
  - What I understood: Documenting blockers explicitly (rather than just noting things as "pending") is important for sprint visibility — it signals to the team that progress is blocked on an external dependency, not on missing effort
